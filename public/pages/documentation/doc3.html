REST API conventions

In REST, everything is treated as a resource, which is defined by a unique URL. Resources can represent entities, such as users and products. There are conventions for the URL path of resources, depending on the HTTP methods used.
Firstly, it is common to use '/api' for an application programming interface URL.
Secondly, for versioning purposes and backwards compatibility, using '/v1' etc. with the version number of the API is common practice. This ensures that anyone using the API will not have their applications broken by updates to the API.
Thirdly, when naming resources, it's a convention to use plural nouns for resource names to indicate you are working with a collection of items. For example '/users' instead of '/user'.
A full CRUD of users could look like this:

HTTP Method Endpoint Description
GET /api/v1/users Returns all resources
GET /api/v1/users/{id} Returns a single resource by id
POST /api/v1/users Create a new resource
PATCH /api/v1/users/{id} Update parts of the resource by id
PUT /api/v1/users/{id} Update the whole resource by id
DELETE /api/v1/users/{id} Deletes a resource by id
